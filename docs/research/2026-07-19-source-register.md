# Reconciliation Architecture — Source Register

> Research date: 2026-07-19. This register distinguishes normative or supervisory sources from public vendor material. Vendor claims identify common product patterns; they are not evidence that a design is compliant or secure.

| ID | Authority / source | Type | What it substantiates | Use and limitation |
|---|---|---|---|---|
| S01 | [ISO 20022 — Bank-to-Customer Cash Management catalogue](https://www.iso20022.org/catalogue-messages/iso-20022-messages-archive?page=1) | Industry message standard | `camt.052` account report, `camt.053` statement, and `camt.054` debit/credit notification are standard cash-management message families. | Use for source adapters and canonical mapping; a bank’s implementation guide still controls its actual payload. |
| S02 | [ISO 20022 — camt.053 definition](https://www.iso20022.org/sites/default/files/documents/messages/mdr_part_2/ISO20022_MDRPart2_BankToCustomerCashManagement_2018_2019_v1_0.pdf) | Industry message standard | A statement reports booked entries and balances and may include underlying transaction detail. | Supports the source-evidence and balance-control design. |
| S03 | [Swift CBPR+ ISO 20022 usage guidance announcement](https://www.swift.com/news-events/news/updated-iso-20022-usage-guidelines-cross-border-payments-released) | Industry infrastructure | Structured payment data, including `camt.053`/`.054`, enables reconciliation automation and cash visibility. | Supports retaining identifiers and structured remittance data. |
| S04 | [Swift cash-management compatibility criteria](https://www2.swift.com/knowledgecentre/rest/v1/publications/s_comp_app_csh_mgt_corp_lbl_crtria_2024/1.0/s_comp_app_csh_mgt_corp_lbl_crtria_2024.pdf?logDownload=true) | Industry infrastructure | Technical reconciliation must capture transport notifications and support error handling, repair, and retransmission. | Supports separate technical and business reconciliation states. |
| S05 | [Swift settlement and reconciliation](https://www.swift.com/securities/settlement-and-reconciliation) | Industry infrastructure | Securities reconciliation spans instructions, movement, holdings, status, custodians, CSDs, CCPs, and T+1/T+0 pressures. | Supports a securities domain overlay; it is not a data-model specification. |
| S06 | [Basel Framework SRP36 — risk data aggregation and reporting](https://www.bis.org/basel_framework/chapter/SRP/36.htm) | Supervisory standard | Governance, robust data architecture, accuracy/integrity, completeness, timeliness, adaptability, reconciliation, validation, exception reporting, documented manual workarounds, and a concept dictionary. | Primary design anchor for a regulated-bank deployment; apply proportionately elsewhere. |
| S07 | [Basel operational-risk principles](https://www.bis.org/basel_consolidated_guidelines/chapter/ORR/10.htm) | Supervisory standard | Regular transaction/account reconciliation is part of operational-risk management. | Establishes reconciliation as a control, not merely productivity software. |
| S08 | [Basel FX-settlement guidance](https://www.bis.org/basel_consolidated_guidelines/chapter/RMA/20.htm) | Supervisory standard | A receipt is not assumed settled until correspondent credit is confirmed and the trade is reconciled; failed transactions require monitoring. | Supports explicit settlement/finality states and timely exception escalation. |

Jurisdictional applicability varies. |
| S10 | [PCAOB AS 2201](https://pcaobus.org/oversight/standards/auditing-standards/details/AS2201) | Audit standard | Effective ICFR gives reasonable assurance over reliable external financial reporting. | Supports testable control design and evidence retention; does not prescribe a particular reconciliation product. |
| S11 | [NIST Cybersecurity Framework 2.0](https://tsapps.nist.gov/publication/get_pdf.cfm?pub_id=957258) | Security framework | Cybersecurity outcomes are organised under Govern, Identify, Protect, Detect, Respond, and Recover. | Framework for the security operating model; not a certification or a product checklist. |
| S12 | [NIST SP 800-53 Rev. 5](https://www.nist.gov/publications/security-and-privacy-controls-information-systems-and-organizations-0) | Security-control catalogue | Controls cover access control, audit/accountability, integrity, availability, and privacy. | Select controls through a risk assessment; do not claim compliance merely by citing it. |
| S13 | [PCI DSS overview](https://www.pcisecuritystandards.org/standards/pci-dss/) | Payment-card security standard | PCI DSS applies to entities that store, process, transmit, or can affect the security of cardholder data. | Scope card-related adapters and data stores correctly; minimize CDE reach. |
| S14 | [PCI DSS logging FAQ](https://www.pcisecuritystandards.org/faqs/1081/) | Payment-card security guidance | Logging must reconstruct who did what, where, when, and how. | Supports audit-event design; exact scope depends on the environment. |
| S15 | [GDPR, Article 5](https://eur-lex.europa.eu/legal-content/EN/TXT/?toc=OJ%3AL%3A2016%3A119%3AFULL&uri=uriserv%3AOJ.L_.2016.119.01.0001.01.ENG) | Privacy law | Purpose limitation, data minimisation, storage limitation, accuracy, integrity, and confidentiality. | Apply where GDPR is applicable; get legal advice for jurisdiction and retention schedules. |
| S16 | [OWASP ASVS](https://owasp.org/www-project-application-security-verification-standard/) | Secure-development verification standard | A testable baseline for web-application technical security controls. | Use as a release gate tailored to the deployment. |
| S17 | [OWASP logging guidance](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html) | Secure-development guidance | Application-level security logging is distinct from web-server logs. | Supports structured domain audit events and redaction. |
| V01 | [BlackLine Transaction Matching API overview](https://developer.blackline.com/docs/matching/1/overview) | Vendor public documentation | Common transaction-matching domains include bank, card, intercompany, and invoice-to-PO matching. | Market-pattern evidence only. |
| V02 | [BlackLine financial close overview](https://www.blackline.com/products/financial-close/) | Vendor public documentation | Production suites integrate high-volume ingestion, configurable matching, exceptions, account reconciliations, journals, approvals, and audit trails. | Market-pattern evidence only; performance claims are not independently verified here. |
| V03 | [Trintech Cadency Match](https://www.trintech.com/cadency/match/) | Vendor public documentation | Multiple match rules, exception management throughout close, manual intervention, and rule-refinement feedback are standard product capabilities. | Market-pattern evidence only. |
| V04 | [AutoRek platform overview](https://autorek.com/autorek) | Vendor public documentation | A common lifecycle is ingestion, data management, matching, exceptions, attestation, journalling, reporting, and audit. | Market-pattern evidence only. |
| V05 | [Duco reconciliation overview](https://du.co/product/reconciliation/) | Vendor public documentation | Normalisation, validation, reconciliation, workflow, and an auditable end-to-end lifecycle are common financial-market patterns. | Market-pattern evidence only. |

## Reliability rule

For a build decision, prefer S01–S17. Use V01–V05 only to learn which capabilities recur in mature commercial platforms. Validate all regulatory applicability, retention periods, and required certifications with the target institution’s compliance, legal, security, and audit functions.
